## 9장: 값 타입

**[JPA의 데이터 타입]**
1. **엔티티 타입**
    - 식별자가 있고, 식별자로 구분한다.
    - 생명주기가 있다.
    - 참조값을 공유할 수 있다. (공유 참조)
2. **값 타입**
    - 식별자가 없다
    - 엔티티에 생명 주기를 의존한다.
    - 공유하지 않고, 값을 복사해서 사용해야한다.

**[값 타입의 종류]**
1. **기본값 타입**: 자바가 제공하는 기본 데이터 타입(int, String)
2. **임베디드 타입**: JPA에서 사용자가 직접 정의한 값 타입
3. **컬렉션 값 타입**: 하나 이상의 값 타입을 저장할 때 사용

## 0. 불변 객체

공유 참조로 인해 예상하지 못한 값의 수정이 발생할 수 있다.
따라서 값 타입은 될 수 있으면 **불변 객체**로 만들어야한다.

- 불변 객체의 참조값 공유는 피할 수 없지만, 인스턴스의 값을 수정할 수 없다 (조회 가능, 수정 불가능)
- 자바의 대표적인 불변 객체: Integer, String

### (1) 설정 방법

생성자로만 값을 설정하고 @Setter(수정자)를 만들지 않으면 된다.
값을 수정하기를 원하면 새로운 객체를 사용해야한다.


## 1. 기본값 타입

- 절대 공유되지 않는다.
*Integer, String은 객체지만 자바 언어에서 기본 타입처럼 사용할 수 있게 지원하므로 기본값 타입으로 지정함*

## 2. 임베디드 타입 (복합 값 타입)

사용자가 직접 정의할 수 있는 새로운 **값** 타입

- 임베디드 타입은 재사용 가능, 높은 응집력을 가진다.
- 엔티티의 생명주기에 의존하므로 UML로 표현하면 composition관계가 된다.
- null이면 매핑한 컬럼값은 모두 null이 된다.

### (1) 테이블 매핑
임베디드 값은 엔티티의 값일 뿐이므로, 값이 속한 엔티티의 테이블에 매핑한다.

### (2) 설정 방법
    아래 두가지 방법 중 하나는 생략할 수 있다.
    1. @Embeddable: 값 타입을 정의하는 곳에 표시
    2. @Embedded: 값 타입을 사용하는 곳에 표시

```java
@Entity
public class Member{
    @Id @GeneratedValue
    private Long id;
    private String name;

    @Embedded Period workPeriod; // 근무 기간
    @Embedded Address homeAddress; // 집 주소
}
```

```java
@Embeddable
public class Period{
    // 기본 생성자가 필수이다.
    @Temporal(TemporalType.DATE) java.util.Date startDate;
    @Temporal(TemporalType.DATE) java.util.Date endDate;

    public boolean isWork(Date date){
        // Period 값 타입을 위한 메서드를 정의할 수 있다.
    }
}
```

```java
@Embeddable
public class Address{
    @Column(name="city") // 해당 값 타입을 사용하는 테이블의 컬럼에 매핑 가능
    private String city;
    private String street;
    private String zipcode;
}
```

### (3) 연관관계

값 타입을 포함하거나 엔티티를 참조할 수 있다

```java
@Embeddable
public class Address{
    private String city;
    private String street;
    @Embedded Zipcode zipcode; // 임베디드 타입 포함 가능
    @ManyToOne PhoneServiceProvide provider // 엔티티 참조 가능
}
```

### (4) 매핑 정보 재정의

`@AttributeOverride`를 통해 한 엔티티에 같은 임베디드 타입을 재정의하여 추가로 사용할 수 있다.

```java
@Entity
public class Member{
    @Id @GeneratedValue
    private Long id;
    private String name;

    @Embedded Address homeAddress; 

    // 같은 Address타입을 추가
    @Embedded 
    @AttributeOverrides({
        @AttributeOverride(
            name = "embedded 타입의 필드명", column=@Column(name="재정의하여 사용할 필드명")
            ),
        @AttributeOverride(...), ... 
    })
    Address companyAddress; 
}
```

## 3. 컬렉션 값 타입

값 타입을 하나 이상 저장하기 위한 타입

- 영속성 전이(Cascade), 고아객체 기능(ORPHAN REMOVE)을 필수로 가진다.

### (1) 설정 방법

- 컬렉션에 보관한다
- `@ElementCollection`, `@CollectionTable` 를 사용한다.

관계형 데이터베이스의 테이블은 컬럼 안에 컬렉션을 포함할 수 없으므로,
별도의 테이블을 추가하고 `@CollectionTable`를 사용하여 추가한 테이블을 매핑해야한다.

```java
@Entity
public class Member{
    ...

    @ElementCollection
    @CollectionTable(name = "FAVORITE_FOODS",
        joinColumns = @JoinColumn(name="MEMBER_ID"))
    @Column(name = "FOOD_NAME")
    private Set<String> favoriteFoods = new HashSet<String>(); // (1)

    @ElementCollection
    @CollectionTable(name = "ADDRESS",
        joinColumns = @JoinColumn(name="MEMBER_ID"))
    private Set<Address> addressHistory = new ArrayList<Address>(); // (2)
}
```

**(1) favoriteFoods는 기본값 타입인 String 컬렉션을 가진다.**
별도의 FAVORITE_FOODS 테이블을 추가하고, `@CollectionTable`을 사용하여 추가한 테이블을 매핑한다.  
이 경우, 값으로 사용되는 컬럼이 하나라면 `@Column`로 컬럼명 지정 가능  

**(2) addressHistory는 임베디드 타입인 Address를 컬렉션으로 가진다.** 
마찬가지로 별도의 ADDRESS 테이블을 사용한다.
테이블 매핑 정보는 `@AttributeOverride`로 재정의 가능하다.

### (2)  값 타입 컬렉션 수정

#### 조회
기본 fetch 전략: LAZY
- 임베디드 값 타입은 엔티티 조회 시 SELECT SQL로 같이 조회 된다.
- 컬렉션 타입은 실제 컬렉션을 사용할 때 SELECT SQL을 1번 조회한다.

#### 수정
1. 임베디드 값 타입 수정: Member 엔티티를 수정하는 것과 같이 동작한다.
2. 기본값 타입 컬렉션 수정: 삭제 후 추가해야한다. 수정할 수 없다.
3. 임베디드 값 타입 컬렉션 수정: 컬렉션에서 삭제 후 새로운 값 등록

### (3) 제약 사항

- **문제 상황:**
    - 값타입 컬렉션에 보관된 값 타입들은 별도의 테이블에 보관된다.
    - 보관된 값 타입의 값이 변경되면 DB의 원본 데이터를 찾기 어렵다

- **JPA 구현체의 해결 방법:** 
    - 값 타입 컬렉션에 변경 사항 발생 시 매핑된 모든 데이터 삭제 후
    - 현재 값 타입 컬렉션 객체에 있는 모든 값을 데이터베이스에 다시 저장한다.

- **실무에서 해결 방법:**
    - 새로운 엔티티를 만들어서 일대다 관계로 설정
    - `영속성 전이 + 고아 객체 기능` 적용 시 값 타입 컬렉션처럼 사용 가능
