# 객체지향 쿼리 언어

## JPQL

> 엔티티 객체를 조회하는 객체지향 쿼리이다.

- JPQL은 객체지향 쿼리이며, 엔티티 객체를 대상으로 쿼리한다.
- SQL을 추상화해서 특정 데이터베이스 SQL에 의존하지 않는다.
- 결국 SQL로 변환된다.

### 장점

- SQL을 추상화해서 특정 데이터베이스에 의존하지 않는다.
    - 데이터베이스 방언만 변경하면 JPQL을 수정하지 않아도 자연스럽게 데이터베이스를 변경할 수 있다.
- JPQL은 SQL보다 간결하다.

### 페치 조인

페치 조인은 JPQL에서 성능 최적화를 위해 제공하는 기능으로 연관된 엔티티나 컬렉션을 한 번에 같이 조회하는 기능이다.

페치 조인은 글로벌 로딩 전략보다 우선한다.

**장점**

- 페치 조인을 사용하면 SQL 한 번으로 연관된 엔티티들을 함께 조회할 수 있어서 SQL 호출 횟수를 줄여 성능을 최적화할 수 있다.
- 페치 조인을 사용하면 연관된 엔티티를 쿼리 시점에 조회하므로 지연 로딩이 발생하지 않는다. 따라서 준영속 상태에서도 객체 그래프를 탐색할 수 있다.

**단점**

- 페치 조인 대상에는 별칭을 줄 수 없다.
- 둘 이상의 컬렉션을 페치할 수 없다.
- 컬렉션을 페치 조인하면 페이징 API를 사용할 수 없다.

## 네이티브 SQL

JPA는 SQL을 직접 사용할 수 있는 기능을 지원하는데 이를 네이티브 SQL이라고 한다.

가끔은 특정 데이터베이스에 의존하는 기능을 사용해야 할 때가 있기 때문에 네이티브 SQL을 사용한다.

네이티브 SQL로 SQL만 직접 사용할 뿐이지 나머지는 JPQL을 사용할 때와 같다. 조회한 엔티티도 영속성 컨텍스트에서 관리된다.

### 단점

- 특정 데이터베이스에 의존하는 SQL을 작성해야 하기 때문에 데이터베이스를 변경하면 네이티브 SQL도 수정해야 한다.

## 벌크 연산

엔티티를 수정할 때 수백 개 이상의 엔티티를 하나씩 처리하기에는 시간이 많이 거린다. 이럴 때는 여러 건을 한 번에 수정하거나 삭제하는 벌크 연산을 사용하면 된다.

다만, 벌크 연산을 사용할 때는 벌크 연산이 영속성 컨텍스트를 무시하고 데이터베이스에 직접 쿼리한다는 점에 주의해야 한다.
