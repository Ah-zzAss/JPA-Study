# 10장: 객체지향 쿼리 언어

## 1. JPQL

- 테이블이 아닌 객체를 대상으로 검색하는 객체지향 쿼리이다.
- SQL을 추상화해서 특정 데이터베이스 SQL에 의존하지 않는다.
- 결국 SQL로 변환된다.

```java
String jpql = "select m from Member m where m.name = :name";

List<Member> result = em.createQuery(jpql, Member.class)
        .setParameter("name", "kim")
        .getResultList();
```

## 2. Criteria와 QueryDsl

JPQL을 편하게 작성하도록 도와주는 빌더 클래스 모음

### (1) 특징

- 문자가 아닌 코드로 JPQL을 작성할 수 있다.
  - 컴파일 시점에 오류 발견 가능
  - IDE를 사용하면 코드 자동완성을 지원
  - 동적 쿼리 작성에 편리하다.
- 메타모델을 이용하여 필드명을 문자로 작성할 수 있다.
  - Criteria는 Member 엔티티 클래스로부터 `Member_`라는 전용 클래스 생성
  - QueryDsl는 Member 엔티티 클래스로부터 `QMember`라는 전용 클래스 생성
- 동적쿼리 사용 가능

> **빌더 클래스**: 객체의 생성 과정과 표현 방법을 분리하여 동일한 생성 절차에서 서로 다른 표현 결과를 만들 수 있도록 하는 생성 디자인 패턴  
> **메타모델**: 어노테이션 정보를 분석하여 생성되는 클래스

### (2) Criteria

JPA에서 제공하는 표준 쿼리 빌더 API이다.

**장점**

- JPQL을 문자열이 아니라 자바 코드로 작성할 수 있다.
- 문법 오류를 컴파일 시점에 어느 정도 확인할 수 있다.
- 동적 쿼리를 작성할 때 유용하다.
- 메타모델을 이용하여 필드명을 문자 대신 코드로 작성할 수 있다.

**단점**

- 코드가 복잡하고 가독성이 떨어진다.

```java
CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<Member> cq = cb.createQuery(Member.class);

Root<Member> m = cq.from(Member.class);

cq.select(m)
  .where(cb.equal(m.get("name"), "kim"));

List<Member> result = em.createQuery(cq).getResultList();
```

### (3) QueryDsl

JPQL을 자바 코드로 작성할 수 있게 도와주는 오픈소스 쿼리 빌더이다.

- Criteria보다 문법이 직관적이고 가독성이 좋다.
- 데이터 조회 기능에 특화되어 있다.
- 동적 쿼리를 작성하기 편리하다.
- 엔티티 클래스를 기반으로 생성된 `Q타입`을 사용한다.
  - 예: `Member` 엔티티 → `QMember`
- 컴파일 시점에 오류를 발견할 수 있고, IDE 자동완성도 지원된다.
- JPA 표준은 아니지만 실무에서 자주 사용된다.

- **프로젝션**: select 절에 조회 대상을 지정하는 것
- **Tuple**: 여러 필드 조회 시 사용하는 내부 타입
- **메서드 위임**: 조건식을 메서드로 분리 가능

```java
QMember member = QMember.member;

List<Member> result = queryFactory
        .selectFrom(member)
        .where(member.name.eq("kim"))
        .fetch();
```


## 3. JDBC 직접 사용, 마이바티스 같은 SQL 매퍼 프레임워크 사용

JDBC 커넥션에 직접 접근하고 싶다면, JPA는 JDBC 커넥션을 휙득하는 API를 제공하지 않으므로  
JPA 구현체(하버네이트)가 제공하는 방법을 사용하여 직접 JDBC 커넥션을 휙득해야한다.

- JDBC, 마이바티스를 JPA와 함꼐 사용하면 영속성 컨텍스트를 적절한 시점에 강제 플러시해야한다.
- JPA를 우회하여 접근하는데, JPA는 전혀 인식을 못한다.
- 이를 해결하기 위해 JPA를 우회해서 SQL실행 직전에 영속성 컨텍스트를 수동으로 플러시하여 동기화해야한다.
- JDBC API: 단순히 데이터의 나열을 조회

```java
em.flush();

Session session = em.unwrap(Session.class);

session.doWork(connection -> {
    PreparedStatement ps =
        connection.prepareStatement("select * from member");
});
```

## 4. Native SQL

JJPQL이 자동 생성하는 SQL을 수동으로 직접 정의하는 것이다.  
JPA가 제공하는 기능 대부분을 그대로 사용할 수 있다.

- 특정 데이터베이스에 종속적인 기능이 필요할 때 사용한다, 이식성이 떨어진다.
- 장점: 엔티티 사용 가능, JPA가 지원하는 영속성 컨텍스트 기능 사용 가능
- SQL 만 직접 사용할 뿐 나머지는 JPQL을 사용할 때와 동일
- JPQL을 사용할 때와 마찬가지로 Query, TypeQuery(Named Native SQL를 사용할 경우에만)를 반환한다.  
  → 따라서 JPQL API를 그대로 사용 가능하다.

```java
String sql = "select * from member where name = ?";

List<Member> result = em.createNativeQuery(sql, Member.class)
        .setParameter(1, "kim")
        .getResultList();
```

### 값 조회

엔티티로 조회하지 않고, 값을 조회하면 스칼라값을 조회했을 뿐이므로 결과를 영속성 컨텍스트가 관리하지 않는다.  
(JDBC가 데이터를 조회한 것과 비슷하다)
```java
List<Object[]> result =
    em.createNativeQuery("select id, name from member")
      .getResultList();
```

### 결과 매핑 사용

엔티티, 스칼라값을 함께 조회하는 것처럼 매핑이 복잡해지면 결과 매핑을 사용해야한다.

```java
@SqlResultSetMapping(
    name = "MemberWithOrderCount",
    entities = @EntityResult(entityClass = Member.class),
    columns = @ColumnResult(name = "order_count")
)
```

### Named Native SQL

미리 이름을 붙여 정의해두고 사용하는 Native SQL이다.

- JPQL의 Named Query처럼 이름을 부여해두고 재사용할 수 있다.
- 애플리케이션 로딩 시점에 SQL 문법을 어느 정도 검증할 수 있다.
- `@NamedNativeQuery` 어노테이션을 사용하여 정의할 수 있다.
- 사용할 때는 `createNamedQuery()` 메소드를 사용한다.
- 결과 타입을 지정하면 `TypedQuery`를 사용할 수 있다.
- Native SQL이므로 특정 데이터베이스에 종속될 수 있어 이식성이 떨어진다.

```java
@NamedNativeQuery(
    name = "Member.findByName",
    query = "select * from member where name = ?",
    resultClass = Member.class
)
@Entity
public class Member {
    // ...
}
```

```java
TypedQuery<Member> query =
    em.createNamedQuery("Member.findByName", Member.class);

List<Member> result = query
        .setParameter(1, "kim")
        .getResultList();
```

## 5. 스토어드 프로시저

- 복잡한 SQL 로직을 DB 내부에 저장해두고 호출할 수 있다.
- IN 파라미터와 OUT 파라미터를 사용할 수 있다.
- 예를 들어 입력값을 두 배로 증가시켜 반환하는 `proc_multiply` 프로시저가 있을 수 있다.

```java
StoredProcedureQuery query =
    em.createStoredProcedureQuery("proc_multiply");

query.registerStoredProcedureParameter(1, Integer.class, ParameterMode.IN);
query.registerStoredProcedureParameter(2, Integer.class, ParameterMode.OUT);

query.setParameter(1, 10);
query.execute();

Integer result = (Integer) query.getOutputParameterValue(2);
```

## 6. 객체 지향 쿼리 심화

### (1) 벌크연산

다음과 같은 방식이 시간이 너무 오래 걸리므로 한 번에 수정/삭제하는 벌크 연산을 사용한다.

- 엔티티를 수정하려면 영속성 컨텍스트의 변경 감지 기능 / 병합을 사용한다.
- 엔티티를 삭제하려면 EmtityManager.remove() 메소드를 사용한다.

**사용방법**

- 수정/삭제 벌크 연산에 executeUpdate 메서드를 사용
- 벌크 연산으로 영향을 받은 엔티티 건수를 반환한다.
```java
int count = em.createQuery(
    "update Member m set m.age = 20")
    .executeUpdate();
```

**문제점**

- 영속성 컨텍스트와 3차 캐시를 무시하고 데이터베이스에 직접 쿼리하므로 조회 시 영속성 컨텍스트에 업데이트 전 데이터가 조회될 수 있다.

**해결방법**

- 가능한 벌크 연산을 먼저 수행하는 것이 좋다.
- 상황에 따라 영송석 컨텍스트를 초기화하는 것이 필요하다.
- 혹은 em.refresh()를 통해 데이터베이스에서 특정 엔티티를 다시 조회한다.

### (2) JPQL과 영속성 컨텍스트

JPQL의 조회 대상은 엔티티, 임베디드 타입, 값 타입이 있다.

- 엔티티를 조회하면 영속성 컨텍스트에서 관리된다.
- 엔티티가 아닌 값 타입은 영속성 컨텍스트에서 관리되지 않는다.

#### JPQL로 조회한 엔티티

JPQL을 통해 엔티티를 조회하면 데이터베이스에 SQL을 실행한다.  
그 후 조회 결과가 이미 영속성 컨텍스트에 존재하는지 확인한다.

이미 같은 식별자의 엔티티가 영속성 컨텍스트에 있으면 기존 엔티티를 반환한다.  
이때 데이터베이스에서 새로 조회한 값은 버린다.

- 새로 검색한 엔티티를 버리는 이유: 영속 상태에 수정 중인 데이터가 사라질 수 있기 때문이다.
- 영속성 컨텍스트는 엔티티의 동일성을 보장한다.

#### find() vs JPQL

- `em.find()`: 엔티티를 영속성 컨텍스트에서 먼저 찾고 없으면 데이터베이스에서 찾는다.
- **JPQL**: 항상 데이터베이스에 SQL을 실행하여 결과를 조회한 뒤, 같은 엔티티가 영속성 컨텍스트에 있으면 기존 엔티티를 반환한다.

```java
Member member = em.find(Member.class, 1L);
```

```java
Member member = em.createQuery(
        "select m from Member m where m.id = :id", Member.class)
    .setParameter("id", 1L)
    .getSingleResult();
```

### (3) JPQL과 플러시 모드

**플러시**: 영속성 컨텍스트의 변경 내역을 데이터베이스에 동기화 하는 것

**플러시 호출 방법**

1. `em.flush()`를 직접 사용
2. 플러시 모드 설정을 통해 커밋 하기 직전 / 쿼리 실행 직전에 자동으로 플러시 호출

**플러시 모드**

- `FlushModeType.AUTO`: 기본값, 트랜잭션 커밋 직전이나 쿼리 실행 직전에 자동으로 플러시 호출
- `FlushModeType.COMMIT`: 커밋 시에만 플러시 호출  
  → 이 옵션은 성능 최적화를 위해 꼭 필요할 때만 사용

  ```java
em.setFlushMode(FlushModeType.AUTO);
em.setFlushMode(FlushModeType.COMMIT);
```
