## 4장: 엔티티 매핑

## 1. 개요
JPA는 자바의 ORM (객체와 관계형 데이터 베이스를 매핑) 기술 표준이다.
즉, JPA를 사용하는데 가장 중요한 일은 엔티티와 테이블을 매핑하는 것이다.

매핑 정보는 annotation, xml 중 선택하여 사용 가능하다.  
annotation이 더 직관적이므로, 이에 관하여서만 설명함)

## 2. 엔티티 매핑

### (1) 매핑 애노테이션

- @Entity : 테이블과 매핑할 클래스에 붙인다. JPA는 이 클래스를 관리한다. 엔티티라고 부른다.
- @Table : 
    - 기본적으로 JPA는 엔티티 이름을 테이블 이름으로 사용하는데, 이 annotation을 통해 매핑할 테이블을 지정할 수 있다.
    - 매핑할 테이블명, 매개변수로 uniqueConstraints 속성 등을 전달하여 제약 조건을 추가할 수 있다.
- @Column : 속성값을 통해 매핑할 컬럼명, 특정 컬럼의 제약 조건을 따로 지정할 수 있다. 

> spring jpa 환경에서는 CamelCase를 snake_case로 자동 변환해주는 전략이 기본으로 적용되어 있다.

### (2) DDL

Table의 uniqueConstraints 속성, @Column 같은 기능들은 단지 DDL을 자동 생성할 때만 사용되고 JPA의 실행로직에는 영향을 주지 않는다.  
따라서 스키마 자동 생성 기능을 사용하지 않고 직접 DDL을 만든다면 불필요하다.  
다만, 제약 조건 파악이 쉽다는 장점때문에 사용하기도함.

### (3) 데이터베이스 스키마 자동 생성

JPA는 데이터 베이스 스키마를 자동으로 생성하는 기능을 지원한다. 

- 애플리케이션 실행 시점에 데이터베이스 테이블이 자동 생성되므로 직접 테이블을 생성할 필요 없다

## 3. 기본키

데이터베이스마다 기본키를 생성하는 방식이 서로 다르다.
JPA의 기본키 생성 전략은 직접할당, 자동 생성 방식으로 나뉜다.

###  (1) 직접 할당
기본키를 application에서 직접 할당한다.
    ```java
    user.setId(1);
    ```

### (2) 자동 생성:
IDENTITY, SEQUENCE 방식은 사용하는 데이터베이스에 의존하는 방법이다. 
@Id에 @GeneratedValue를 추가하고 다음 중 원하는 자동 생성 전략을 선택한다.

#### IDENTITY 전략
기본 키 생성을 DB에 위임한다.

- 데이터베이스에 값을 저장하고 나서야 기본 키 값을 구할 수 있을때 사용
- 쓰기 지연이 동작하지 않는다: `em.persist()`를 호출하면 바로 쓰기 지연 저장소의 SQL문이 flush 되므로

1. `em.persist()`를 호출
2. 즉시 DB에 INSERT SQL을 전송 (쓰기 지연 SQL 저장소를 거치지 않음)
3. 데이터 베이스에서 생성한 식별자를 반환받는다.
4. 반환받은 식별자를 entity의 @Id 필드에 채워넣고 영속화한다.

#### SEQUENCE 전략
데이터베이스 시퀀스를 사용해서 기본키를 할당한다.

- DB 시퀀스를 통해 식별자를 조회하는 추가작업이 필요하므로 DB와 2번 통신한다.
    > 1. 식별자를 구하려고 DB 시퀀스를 조회한다.
    > 2. 조회한 식별자를 pk로 사용하여 DB에 저장한다.
- 최적화 방법은 @SequenceGenerator.allocationSize를 한 번에 증가시키고 나서 그만큼 메모리에 시퀀스 값을 할당한다.  
이 방식은 시퀀스 값을 선점하므로 여러 JVM에서 동시 동작해도 pk 값이 충돌하지 않는 장점이 있다.  -> 왜? DB에서도 어차피 하나씩 select해와도 충돌 안하지 않나?
    > 예를 들어 50 정도를 allocationSize 값으로 지정하면,
    > 시퀀스가 한 번에 50이 증가 -> 메모리에서 1~50까지는 시퀀스 값을 할당 -> 51이 되면 다시 시퀀스 값을 100으로 증가...

1. `em.persist()`를 호출
2. 데이터베이스의 시퀀스를 활용해서 식별자를 조회한다.  
JPA가 DB의 시퀀스 객체에 `SELECT nextval for sequence_name` 쿼리를 날림
3. DB가 다음 식별자 값을 반환함
4. 조회한 식별자를 entity의 id값에 할당한 후, 영속성 컨텍스트에 저장한다.
5. 트랜잭션을 커밋하여 flush가 일어나면 INSERT 쿼리를 날려서 DB에 저장

#### TABLE 전략
키 생성용 테이블을 하나 만들어두고 시퀀스처럼 사용하는 방법이다.
테이블을 활용하는 방식으로 모든 데이터베이스에서 사용 가능하다.

커스텀 시퀀스 테이블을 사용한다는 것만 제외하면 SEQUENCE 전략과 내부 동작 방식이 같다.

```
CREATE table MY_SEQUENCE (
    sequence_name varchar(255) not null # 시퀀스 이름
    next_val bigint # 시퀀스 값
    primary key(sequence_name)
)
```

#### AUTO 전략

선택한 DB 방언에 따라 위의 세 전략 중 하나를 자동으로 선택한다.
데이터베이스를 변경해도 코드를 수정할 필요가 없다는 장점이 있다.

## 3, 영속성 컨텍스트
영속성 컨텍스트는 엔티티를 식별자 값으로 구분한다.
따라서 엔티티를 영속 상태로 만들기 위해서는 식별자 값이 필요하다.

em.persist() 후 생성 전략마다 발생하는 일은 다음과 같다.
1. 직접 할당: 호출 전 애플리케이션에서 직접 식별자값을 할당하므로 영속 상태로 만들 수 있다.
2. 자동 생성:
    - SEQUENCE: 데이터 베이스 시퀀스에서 식별자 값을 휙득한 후 영속성 컨텍스트에 저장한다.
    - TABLE: 데이터베이스 시퀀스 생성용 테이블에서 식별자 값을 휙득한 후 영속성 컨텍스트에 저장한다.
    - IDENTITY: 데이터베이스에 엔티티를 저장해서 식별자 값을 휙득한 후 영속성 컨텍스트에 저장한다.
