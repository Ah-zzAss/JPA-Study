## 8장: 프록시와 연관관계 관리

## 1. 프록시
**실제 객체를 참조**를 보관하는 객체이다.

### (1) 특징
- 처음 사용할 때 한 번만 초기화 된다.
- 프록시 객체가 초기화 되면 프록시 객체가 참조하는 주소를 통해 실제 엔티티에 접근 가능하다.
- 프록시 객체는 원본 엔티티를 상속받은 객체이므로 타입 체크 시 주의하여 사용해야한다.
- 프록시 객체의 초기화는 영속성 컨텍스트가 담당한다. 따라서 준영속 상태의 프록시를 초기화하면 문제가 발생한다.

### (2) 초기화 과정
1. 프록시 객체에 메서드를 호출한다.  
(실제 엔티티가 영속성 컨텍스트에 생성되어 있으면 해당 엔티티를 반환한다.)
2. 영속성 컨텍스트에 실제 엔티티 생성을 요청한다.
3. 영속성 컨텍스트는 데이터베이스를 조회하여 실제 엔티티 객체를 생성한다.
4. 프록시 객체는 생성된 실제 엔티티 객체의 참조를 보관한다.  (실제 엔티티 객체는 영속성 컨텍스트에서 관리되는 상태)
5. 프록시 객체는 실제 엔티티 객체의 메서드를 호출하여 반환한다.

### (3) 식별자 값을 가진다
프록시 객체는 식별자 값을 가지고 있다.

- AccessType.PROPERTY (Getter에 @Id가 있는 경우)일 때 getId()를 호출해도 프록시를 초기화하지 않습니다.
- AccessType.FIELD로 설정하면 .getId()가 어떤 일을 하는 메서드인지 알지 못하므로 프록시 객체를 초기화 합니다.
- 연관관계 설정 시 식별자 값만 사용하므로 프록시를 이용하면 데이터베이스 접근 횟수를 줄일 수 있음 (이때는 엔티티 접근방식이 FIELD여도 프록시를 초기화하지 않음)

### (4) 컬렉션 래퍼: 컬렉션에 대한 프록시
하버네이트는 엔티티를 영속상태로 만들 때, 엔티티에 컬렉션이 있으면 추척/관리할 목적으로 **원본 컬렉션을 내장 컬렉션으로 변경**된 것
- 컬렉션에 대한 프록시 역할을 한다.
- `member.getOrders()`을 호출해도 컬렉션은 초기화 되지 않고 `member.getOrders().get(0)`처럼 실제 데이터를 조회할 때 된다.

### (4) 프록시 확인
- 프록시 인스턴스의 초기화 여부: `PersistenceUnitutil.isLoaded(Object entity)`
    - true: 이미 초기화 or 프록시 인스턴스가 아닌 경우
    - false: 초기화 전

- 조회한 엔티티가 프록시인지 진짜 엔티티인지: `클래스.getClass().getName()`로 클래스명 직접 출력하여 확인  
    - 클래스명 뒤가 `.javassist`라면 프록시이다.

## 2. 지연 로딩 / 즉시 로딩

**[JPA의 기본 전략]**
- **즉시 로딩**: 연관된 엔티티가 하나
- **지연 로딩**: 연관된 엔티티가 컬렉션

개발 시 추천하는 방식은 모든 연관 관계에 지연 로딩 사용 후 실제 사용 상황을 파악 후 필요한 곳에만 즉시 로딩 사용

### (1) 즉시 로딩 (Eager Loading)
엔티티를 조회할 때 연관된 엔티티도 함께 조회한다.

- 설정 방법: `@ManyToOne(fetch = FetchType.EAGER)`
- 최적화: JPA 구현체는 즉시 로딩을 하기 위해 조인 쿼리를 사용한다.  
    > 선택적 관계라면 OUTER JOIN, 필수 관계라면 INNER JOIN을 사용한다.  
    (@JoinColumn(nullable= false)를 통해 외래키가 null값을 허용하지 않는다고 나타내어야한다.)
- 주의점:
    - 컬렉션을 하나 이상 즉시 로딩하는 것은 권장하지 않는다.
    - 컬렉션 즉시 로딩은 항상 OUTER JOIN을 사용한다.

### (2) 지연 로딩
연관된 엔티티를 실제 사용할 때 조회한다.

- 설정 방법: `@ManyToOne(fetch = FetchType.LAZY)`

**지연 로딩 구현 방법:**
1. 프록시를 사용하는 방식
2. 바이트코드를 수정하는 방식 (설정이 복잡하므로 설명 생략)

> JPA의 표준 명세는 지연 로딩의 구현 방법을 구현체에 위임하였다.
> 따라서 해당 방식은 하버네이트 구현체에 대한 내용이다.

## 3. 영속성 전이: CASCADE

특정 엔티티를 만들 때 **연관된 엔티티를 함께 영속 상태로** 만들기 위해 사용하는 기능

```java
public enum CascadeType{
...
}
```
- **여러 속성 사용 가능:** cascade = {CascadeType.PERSIST, }
- `CascadeType.PERSIST`, `CascadeType.REMOVE`는 `em.persist(), em.remove()` 실행 시 바로 전이가 발생하지 않고, **플러시를 호출할 때 전이**가 발생


### (1) 저장
(부모를 영속화할 때 연관된 자식들도 함께 영속화하는 예제)

JPA에서 엔티티 **저장 시 연관된 엔티티는 모두 영속 상태여야한다.**
`cascade = CascadeType.PERSIST` 옵션을 사용하면 **부모만** 영속화하면 **자식도 함께 영속화**할 수 있다.

### (2) 삭제

**부모 엔티티만 삭제하면** 연관된 자식 엔티티도 **함께 삭제** 가능

```java
Parent parent = em.find(Parent.class, 1L);
em.remove(parent); 
```

- 코드 실행 시: DELETE SQL문 3번 + 외래키 제약 조건을 고려하여 자식 먼저 삭제 후 부모 삭제


## 4. 고아 객체 제거
**고아 객체** (부모 엔티티와 연관관계가 끊어진 자식 엔티티) 를 **자동으로 삭제**하는 기능

- 설정 방법: `@OneToMany(mappedBy = "parent", orphanRemoval = true)`
- 주의: **참조하는 곳이 하나일 때**만 사용해야하므로 **@OneToOne, @OneToMany에만 사용 가능**  
(나머지 연관관계에는 지원 자체를 안함)

- **기능**:
    1. 부모 엔티티의 컬렉션에서 **자식 엔티티의 참조만 제거**하면 자식 엔티티가 자동으로 삭제
    2. 개념적으로 부모 제거 시 자식이 고아가 되므로, **부모를 제거하면** 자식이 같이 제거된다. (CascadeType.REMOVE와 동작이 같다.)

### 5. 생명 주기 관리

엔티티는 스스로 생명 주기를 관리한다.
즉, 일반적으로 엔티티는 `EntityManager.persist()` 를 통해 영속화, `EntityManager.remove()` 를 통해 제거된다.

- `CascadeType.All + orphanRemoval = true` 를 동시에 사용하면 **부모 엔티티를 통해 자식의 생명주기를 관리할 수 있다.**
- 자식을 저장하려면 부모의 자식 속성에 자식을 등록하여 부모 엔티티만 `persist()` 하면 된다.
- 자식을 삭제하려면 부모의 자식 속성에서 참조를 제거하면 된다.
