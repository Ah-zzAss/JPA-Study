## 6장: 다양한 연관관계 매핑

연관관계 매핑 시 고려할 점으로는 세가지가 존재한다.

**1. 다중성**

- 다대일(N:1)
- 일대다(1:N)
- 일대일(1:1)
- 다대다(N:M)

> - 앞쪽에 위치한 것이 연관관계 주인이라고 가정하고 설명
> - 다대다는 실무에서 직접 사용을 지양, 보통 연결 엔티티를 추가하여 일대다·다대일 관계로 풀어서 사용

**2. 방향**

테이블에는 외래키 하나로 JOIN하여 쿼리가 가능하므로 방향이라는 개념이 존재하지 않는다.  

반면 객체 관점에서는 다음과 같이 분리한다.
- 단방향: 한 쪽만 참조
- 양방향: 서로 참조

**3. 연관관계의 주인**

외래키를 직접 관리하는 쪽, 주인이 아닌 쪽은 읽기만 가능하다.

## 1. 다대일(@ManyToOne)

**예시: 다대일 관계인 Member와 Team**
- DB 테이블: 하나의 팀은 0명 이상의 멤버를 가지고, 멤버는 하나의 팀에 속할 수 있다.

### (1) 단방향

```java
@Entity
public class Team{
    ...

    @Id @GeneratedValue
    @Column(name = "TEAM_ID")
    private Long id
}

@Entity
public class Member{
    ...

    @ManyToOne
    @JoinColumn(name="TEAM_ID")
    private Team team
}
```

### (2) 양방향
- 주인이 아닌 Team.members는 조회를 위한 JPQL이나 객체 그래프 탐색에 사용한다.  
    (일대다, 다대일 연관관계에서는 DB 테이블 구조에서 항상 '다'쪽에 외래키가 존재하고, JPA는 외래키를 관리할 때 연관관계의 주인만 사용한다.)
- 양방향 연관관계는 항상 서로를 참조해야한다.
    - 항상 서로를 참조하게 하기 위해서는 `setTeam`, `addMember`와 같은 편의 메서드를 작성하는 것이 좋다
    - 양방향 연관관계는 항상 무한루프를 조심해야한다.

```java
@Entity
public class Member {

    @ManyToOne
    @JoinColumn(name = "TEAM_ID")
    private Team team;

    public void setTeam(Team team) {
        this.team = team;

        if (team != null && !team.getMembers().contains(this)) {
            team.getMembers().add(this);
        }
    }
}

@Entity
public class Team {

    @Id
    @GeneratedValue
    @Column(name = "TEAM_ID")
    private Long id;

    @OneToMany(mappedBy = "team")
    private List<Member> members = new ArrayList<>();

    public void addMember(Member member) {
        members.add(member);
        if (member.getTeam() != this) { // 무한 루프 방지
            member.setTeam(this);
        }
    }

    public List<Member> getMembers() {
        return members;
    }
}
```

## 2. 일대다(@OneToMany)

일대다에서 일쪽이 연관관계의 주인인 경우이다.  
이 때 엔티티를 하나 이상 참조 가능하므로 자바 컬렉션을 사용해야한다.

### (1) 단방향
member(1)에서 team(N)의 id를 관리하는 방법, 보통 자신이 매핑한 테이블의 외래키를 관리하는데,  
이 매핑은 반대쪽에 있는 외래키를 관리한다.

- DB 테이블 상 memeber 테이블의 fk인 team_id를 Team 엔티티의 필드가 들고 있음
- **Team 엔티티 입장**에서는 **매핑된 자신의 테이블이 아니라** **다른 테이블의 외래키**를 관리해야한다. -> 관리가 어렵다!
- 일대다 단방향보다는 **다대일 양방향**을 권장

```java
@Entity
public class Team{
    @Id @GeneratedValue
    @Column(name = "TEAM_ID")
    private Long id

    @OneToMany
    @JoinColumn(name="TEAM_ID") // member 테이블의 TEAM_ID
    private List<Member> members = new ArrayList<Member>();
}

@Entity
public class Member{
    ...
    @Id @GeneratedValue
    @Column(name="MEMBER_ID")
    private Long id
}
```

### (2) 양방향

존재하지 않음, 대신 다대일 양방향 매핑을 사용해야함  
일대다는 연관관계의 주인이 될 수 없다.

## 3. 일대일(@OneToOne)

일대일은 주테이블, 대상테이블 어느 곳이나 외래키를 가질 수 있다

### (1) 주테이블에 외래키

일대일 관계 구성 시 객체지향 개발자들은 주 테이블에 외래키가 있는 것을 선호하며,
JPA도 주 테이블에 외래키가 있으면 더 편하게 매핑 가능

- 다음은 'Member는 반드시 하나의 Locker를 가진다' 는 관계를 단방향, 양방향으로 설명한다.

### 단방향

```java
@Entity
public class Member{
    @Id @GeneratedValue
    @Column(name= "MEMBER")
    private Long id;

    @OneToOne // 주 테이블에서 연관관계 설정
    @JoinColumn(name = "LOCKER_ID")
    private Locker locker
}

@Entity
public class Locker{
    @Id @GeneratedValue
    @Column(name= "LOCKER_ID")
    private Long id;
    ...
    // mappedBy로 주인이 아닌쪽을 설정하지 않음 -> 양방향이 아님
}
```

### 양방향

양방향의 경우는 위의 예시에서 주인이 아닌 Locker에 mappedBy를 설정

```java
@Entity
public class Locker{
    @Id @GeneratedValue
    @Column(name= "LOCKER_ID")
    private Long id;

    @OneToOne(mappedBy = "locker")
    private Member member
}
```

### (2) 대상 테이블에 외래키

### 단방향

JPA 에서 미지원

### 양방향

주 엔티티인 Member 대신 Locker를 연관관계 주인으로 만들고, Locker 테이블의 외래키를 관리하도록 함

```java
@Entity
public class Member{
    @Id @GeneratedValue
    @Column(name= "MEMBER_ID")
    private Long id;

    @OneToOne(mappedBy = "member")
    private Locker locker
}

@Entity
public class Locker{
    @Id @GeneratedValue
    @Column(name= "LOCKER_ID")
    private Long id;

    @OneToOne
    @JoinColumn(name = "MEMBER_ID")
    private Member member
}
```

## 4. 다대다(@ManyToMany)

- 다대다 연결일 경우 연결 테이블을 JPA에서 처리해주므로 편하다
- 그러나 연결 테이블에 필드가 추가되면 더는 사용할 수 없으므로 연결 엔티티를 만들어서 일대다, 다대일을 사용하는 것을 지향하자


### (1) 연결테이블 설계 시 식별자 구성

데이터 베이스 설계 시 식별 관계는 두가지로 나뉜다.
- 식별 관계: 받아온 식별자를 기본키 + 외래키로 사용
- 비식별 관계: 받아온 식별자를 외래키로만 사용, 새로운 식별자를 추가

객체 입장에서는 비식별관계를 사용하는 것이 **복합키를 위한 식별 클래스를 만들지 않아도** 되므로 단순하고 편하게 매핑 가능  
(식별 관계는 보통 복합키를 사용해야 하므로 JPA 매핑이 복잡해진다.)
