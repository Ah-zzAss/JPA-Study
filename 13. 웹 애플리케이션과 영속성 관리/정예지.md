## 13장: 웹 애플리케이션과 영속성 관리

## 1. 트랜잭션 범위의 영속성 컨텍스트

- 스프링 컨테이너가 기본으로 사용하는 전략
- **트랜잭션의 범위와 영속성 컨텍스트의 생존 범위가 동일한 전략**
  > - 트랜잭션 시작 시 영속성 컨텍스트 생성
  > - 트랜잭션 종료 시 영속성 컨텍스트 종료

1. **같은** 트랜잭션 → **같은** 영속성 컨텍스트 사용

2. **다른** 트랜잭션 → **다른** 영속성 컨텍스트 사용
   - 여러 스레드에서 동시에 요청이 들어와도, 같은 엔티티 매니저를 사용하더라도 트랜잭션에 따라 접근하는 영속성 컨텍스트가 다름
   - 스프링 컨테이너가 스레드마다 각각 다른 트랜잭션을 할당함  
     따라서 멀티스레드 상황에서도 같은 엔티티 매니저를 사용해도 접근하는 영속성 컨텍스트가 다르므로 안전함

## 2. 준영속 상태와 지연 로딩

보통 서비스 계층에 트랜잭션을 적용한다.  
따라서 트랜잭션이 끝난 뒤 컨트롤러나 뷰에서는 엔티티가 준영속 상태가 된다.

준영속 상태에서는 다음 기능이 동작하지 않는다.

- 변경 감지
- 지연 로딩

### 문제 상황

뷰를 렌더링할 때 연관된 엔티티도 함께 사용해야 하는데,  
연관 엔티티가 지연 로딩으로 설정되어 있으면 프록시 객체로 조회된다.

하지만 뷰에서는 이미 엔티티가 준영속 상태이므로 지연 로딩을 사용할 수 없다.  
따라서 오류가 발생할 수 있다.

### 해결 방법

### 1. 뷰가 필요한 엔티티를 미리 로딩

엔티티가 준영속 상태가 되기 전에 필요한 연관 엔티티를 미리 로딩해두는 방법이다.  
이렇게 하면 뷰에서 지연 로딩이 발생하지 않는다.

**미리 로딩하는 방법**은 크게 2가지가 있다.

**1. 글로벌 페치 전략 수정**

엔티티의 fetch 타입을 즉시 로딩으로 변경하는 방법이다.

```java
@ManyToOne(fetch = FetchType.EAGER)
```

- **단점**
  - 사용하지 않는 엔티티까지 함께 로딩할 수 있음
  - N+1 문제가 발생할 수 있음 → JPQL 페치 조인으로 해결 가능
    > - 단건 조회할 때는 글로벌 fetch 전략을 참고하여 함께 JOIN하여 가져온다.
    > - JPQL로 조회할 때는 글로벌 fetch 전략을 참고하지 않고, JPQL 자체만 보고 SQL을 생성한다.
    > - 따라서 즉시 로딩으로 글로벌 fetch 전략을 적용해두었더라도 JPQL 사용 시 N+1 문제 발생 가능

**2. JPQL 페치 조인**

JPQL을 호출하는 시점에 함께 로딩할 엔티티를 선택하는 방법이다.

- SQL JOIN을 사용해 페치 조인 대상까지 함께 조회
- N+1 문제를 방지할 수 있음

- **단점**
  - 뷰와 리포지토리 간에 논리적인 의존관계가 생김
  - 화면에 맞춘 리포지토리 메서드가 증가할 수 있음

### 2. OSIV 사용

영속성 컨텍스트를 뷰까지 열어두는 방식으로, **뷰에서도 지연 로딩**을 사용할 수 있다.  

## 3. OSIV

OSIV는 **Open Session In View**의 약자이다.  
영속성 컨텍스트를 뷰까지 열어두는 방식이다.

### 과거 OSIV: 요청 당 트랜잭션 방식

요청이 들어오자마자 트랜잭션을 시작하고, 요청이 끝날 때 트랜잭션을 종료하는 방식이다.

- **단점**: 컨트롤러나 뷰 같은 프레젠테이션 계층에서도 엔티티를 수정할 수 있다.
- **해결 방법**
  - 아래 방법들은 모두 코드량이 증가한다는 단점이 있음
  1. 엔티티를 읽기 전용 인터페이스로 제공
  2. 엔티티 래핑: 엔티티의 읽기 전용 메서드만 가진 객체로 감싸서 반환
  3. DTO만 반환: 단순 데이터 전달 객체인 DTO를 생성해서 반환

### 스프링 OSIV: 비즈니스 계층 트랜잭션

스프링 OSIV는 트랜잭션을 비즈니스 계층에서만 사용한다.

- **동작 원리**
  1. 영속성 컨텍스트는 요청이 들어올 때 생성되고, 요청이 끝날 때 종료된다.
  2. 트랜잭션은 서비스 계층에서만 시작되고 종료된다.

- **특징**
  - 영속성 컨텍스트를 통한 모든 변경은 트랜잭션 안에서 이루어져야 한다.
  - 트랜잭션 없이 영속성 컨텍스트를 변경하고 플러시하면 예외가 발생한다.
  - 트랜잭션 없이 읽기는 가능하다.
    - 트랜잭션 안에서는 엔티티 조회와 수정이 가능하다.
    - 트랜잭션 밖에서는 엔티티 조회만 가능하다.
    - 따라서 프레젠테이션 계층에서도 지연 로딩을 사용할 수 있다.

## 4. 강제 초기화와 FACADE 계층

**영속성 컨텍스트가 살아있을 때** **프레젠테이션 계층에 필요한 엔티티를 강제로 초기화**해서 반환하는 방법이다.  
_(글로벌 fetch 전략은 지연 로딩이라고 가정한다.)_

- 프록시 객체로 조회
- 실제 사용하는 시점에 초기화
- 서비스 계층이 프레젠테이션 계층을 위해 프록시 초기화까지 담당하게 됨
- 이 경우 뷰가 필요한 엔티티에 따라 서비스 계층의 로직을 변경해야 하는 문제가 생김

이를 해결하기 위해 **서비스 계층과 프레젠테이션 계층 사이에 FACADE 계층을 둘 수 있다.**

- **FACADE 계층의 역할**
  - 프레젠테이션 계층에서 필요한 프록시 객체 초기화
  - 서비스 계층을 호출하여 비즈니스 로직 실행
  - 리포지토리를 직접 호출하여 뷰가 요구하는 엔티티 조회
  - 서비스 계층과 프레젠테이션 계층 사이의 논리적인 의존성 분리

- **단점**
  - 중간에 계층이 하나 더 추가됨
