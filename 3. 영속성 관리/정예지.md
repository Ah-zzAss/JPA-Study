## 3장: 영속성 관리

## 1. 개요

### 엔티티 매니저 팩토리
**엔티티 매니저를 만드는** 공장이다.
- 엔티티 매니저 팩토리를 만드는 비용을 크다, 따라서 하나만 생성해서 application 전체에서 공유하도록 설계되어 있다.
- 여러 스레드가 접근, 공유해도 안전하다.

### 엔티티 매니저
엔티티를 관리하는 관리자, **엔티티를 저장하는 가상의 데이터 베이스**이다.

- 여러 스레드가 접근, 공유하면 동시성 문제가 발생한다. (절대 공유 금지)

## 2. 영속성 컨텍스트
**엔티티를 영구 저장하는 환경**이라는 의미이다.

- 엔티티 매니저는 영속성 컨텍스트에 엔티티를 보관하고 관리한다.
- 영속성 컨텍스트에 접근하기 위해서는 **엔티티 매니저를 통해서 접근 가능**하다.

### 영속성 컨텍스트 구성: 
#### (1) 1차 캐시:
- 영속 상태의 엔티티는 모두 이곳에 저장된다.
- 키값은 @Id값, 값은 엔티티 인스턴스의 매핑 쌍으로 저장한다.
- 키값은 DB의 pk와 매핑되어 있다.

#### (2) 쓰기 지연 SQL 저장소:
DB에 명령할 SQL문들을 여기에 모아둔다.

- flush 될 때 쓰기 지연 SQL 저장소에 모인 쿼리들이 DB에 보내진다.
-  `쿼리를 호출할 때마다 DB에 등록 쿼리를 보내고 트랜잭션 커밋 하는 것` 과 `모아둔 등록 쿼리를 DB에 보낸 후 커밋하는 것` 은 결과가 동일하다. 다만, 한 번에 보내는 방식을 잘 활용하면 성능을 최적화할 수 있다.

#### (3) 스냅샷:
엔티티를 **영속성 컨텍스트에 보관할 때 최초 상태**를 복사하여 저장하는 것

- flush될 때 엔티티와 스냅샷을 비교하여 변경된 엔티티를 찾는다.
- **관리를 시작하기 시작한 때의 엔티티 상태**를 저장해둔다.  
    더티 체킹 시에 이 스냅샷과 비교하여 달라진 부분에 대해 UPDATE Query를 쓰지 지연 SQL에 넣어둔다.

### 특징

#### (1) 조회

무조건 JPA는 조회 시 기본적으로 영속성 컨텍스트의 1차 캐시를 확인한다.

- **1차 캐시에 있다면**, 그 캐시된 엔티티를 조회한다.
- **1차 캐시에 없다면**
    1. DB에서 조회하여 엔티티를를 생성하여 1차 캐시에 저장한다.  
    (이 때 스냅샷에 해당 엔티티를 저장한다.)
    2. 1차 캐시에서 조회한영속 상태의 엔티티를 반환한다.

#### (2) 등록

쓰기 지연 SQL 저장소에 INSERT문을, 1차 캐시에는 해당 엔티티를 저장한다.  
flush 될 때 쓰기 지연 SQL 저장소에 모인 쿼리를 DB에 보내고, 이후 트랜잭션이 커밋된다.

#### (3) 수정

JPA는 **변경 감지 (dirty checking)**을 통해 엔티티를 수정한다.

1. 트랜잭션을 커밋하면 먼저 엔티티 매니저 내부에서 flush가 먼저 호출된다.
2. 엔티티와 스냅샷을 비교하여 변경된 엔티티를 찾는다.
3. 변경된 엔티티가 있으면 수정 쿼리를 생성하여 쓰기 지연 SQL 저장소에 넣고 바로 DB로 보낸다.
4. DB 트랜잭션을 커밋한다.

> **수정 쿼리를 생성은 어떻게 될까?**  
> 모든 필드를 업데이트하는 UPDATE SQL문을 사용한다.
>
> - **장점**:  
>    (1) 수정 쿼리가 항상 같다, app 로딩 시점에 수정 쿼리를 미리 생성해두고 재사용 가능  
>    (2) 이전에 한 번 파싱된 쿼리 재사용 가능
> - **단점**:   
>    (1) 데이터 전송량이 증가한다.   
>    (단, 컬럼이 약 30개 이상일 경우에만 부각되는 단점이다, 이 경우 동적 SQL 사용을 위해 @DynamicUpdate 사용 가능)


#### (4) 삭제

1. 키값을 통해 삭제 대상 엔티티를 조회
2. `remove` 메서드를 호출
    - DELETE 쿼리를 쓰기 지연 SQL 저장소에 등록
    - 영속성 컨텍스트에서 바로 제거
3. 트랜잭션을 commit해서 flush를 호출하면 DB에 삭제 쿼리를 전달

## 3. 플러시

**영속성 컨텍스트의 변경 사항을 DB에 동기화**하는 작업 (초기화 작업이 아니다)
따라서 **쓰기 지연 SQL 저장소는 비워지지만, 1차 캐시는 그대로 남는다**

#### flush가 실행되면
1. 변경 감지가 동작해서 영속성 컨텍스트의 모든 엔티티를 스냅샷과 비교
2. 비교하여 찾은 엔티티들의 수정 Query를 만들어 쓰기 지연 SQL 저장소에 등록
3. 쓰기 지연 SQL 저장소의 쿼리를 DB에 전송

#### 영속성 컨텍스트를 flush하는 방법
**1. 직접 호출**  

**2. 트랜잭션 커밋 시 자동 호출**
- 트랜잭션 커밋 전에 flush를 호출하지 않을 경우 DB에 어떤 내용도 반영되지 않음
- JPA는 이러한 문제를 방지하기 위해 트랜잭션 커밋 시 flush 자동 호출

**3. JPQL 쿼리 실행 시 자동 호출**  
JPQL는 SQL로 변환되어 DB에서 엔티티를 조회,  
따라서 해당 쿼리 실행 직전에 영속성 컨텍스트를 flush하여 DB에 반영해야한다

## 4. 준영속

준영속 상태란 **영속 상태였던 엔티티를 엔티티 컨텍스트가 관리하지 않는** 상태  
**1차 캐시, 쓰기 지연 SQL 저장소**에서 **해당 엔티티를 관리하기 위한 정보가 제거**된다.

#### (1) 엔티티를 준 영속 상태로 변환 `detach()`
특정 엔티티 **하나**를 준영속 상태로 변경

#### (2) 영속성 컨텍스트 초기화 `clear()`
영속 상태였던 엔티티를 **모두** 준영속 상태로 변경

- 영속성 컨텍스트는 여전히 존재, 그치만 비어있는 상태

#### (3) 영속성 컨텍스트 종료 `close()`
영속 상태였던 엔티티가 **모두** 준영속 상태로 변경

#### (4) 병합 `merge()`
준영속 상태의 엔티티를 영속 상태로 다시 병합

- 준영속 상태의 엔티티를 받아서 그 정보로 새로운 영속 상태의 엔티티를 반환
- 이 경우 준영속 상태였던 엔티티와 동등성(equals)은 갖지만, 동일성(==)은 갖지 않는다.
    (내부 값은 같지만, 인스턴스 참조값이 다르다)

**다시 병합할 경우**
1. 병합하기 위해 준영속 상태의 엔티티로 merge()를 실행한다.
2. 해당 엔티티의 식별자값으로 1차 캐시에서 다시 조회
    - 보통은 이 경우 준영속 상태이미로 다시 조회할 수 없지만 JPA는 1차 캐시 확인 후 DB 접근을 원칙으로 한다.
3. DB에서 해당 엔티티를 다시 조회
4. 조회한 엔티티를 스냅샷으로 저장
5. 조회한 엔티티를 1차 캐시에 저장
6. 엔티티 반환
