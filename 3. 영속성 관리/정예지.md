## 3장: 영속성 관리

## 1. 개요

엔티티 팩토리 매니저는 엔티티 매니저를 여러개 생성하기 위해 필요하다.
단, 엔티티 팩토리 매니저는 생성하는게 리소스가 많이 들기 때문에 하나 생성해놓고 공유해서 사용한다.
(여러 스레드에서 공유해서 사용해도 괜찮다)

엔티티 매니저는 공유해서 사용할 수 없다 요청 하나 당 하나씩 생성된다.
엔티티 매니저 안에 영속성 컨텍스트라는 논리적 개념이 존재하는데, 이 개념을 사용하여 엔티티들을 관리한다.

엔티티 매니저는 하나의 영속성 컨텍스트를 가진다.
(뒤에 한 엔티티 매니저가 여러 영속성 컨텍스트를 가지는 경우가 있지만 지금은 다루지 않는다)

## 2. 영속성 컨텍스트

영속성 컨텍스트는 엔티티들을 관리한다.

영속성 컨텍스트는 다음과 같은 주요 기능이 있다.

- 1차 캐시: 
관리 중인 객체는 여기에 넣어서 관리한다. 만약에 없는 경우는 DB에서 조회하여 가져온다.
저장할 때는 "@Id값, 객체"의 매핑으로 저장하는데, 여기서 id값은 DB의 pk값과 동일하다

- 쓰기 지연 SQL 저장소:
DB에 명령할 SQL문들을 여기에 모아둔다.
어차피 commit하기 전 한 번에 SQL문을 flush하는 것과 하나씩 SQL문을 flush하고 commit하는 것은 동일하므로, 성능상 유리한 점을 가져갈 수 있는 방법인 전자를 택하는데, 이 SQL문을 모아두는 곳이 쓰기 지연 SQL 저장소다.

- 스냅샷:
관리를 시작하기 시작한 때의 엔티티 상태를 저장해둔다.
더티 체킹 시에 이 스냅샷과 비교하여 달라진 부분에 대해 UPDATE Query를 쓰지 지연 SQL에 넣어둔다.

(1) 조회

1. 무조건 JPA는 조회 시 기본적으로 영속성 컨텍스트의 1차 캐시를 확인한다.
2. 있다면 그 캐시된 엔티티를 반환한다.
3. 없다면 DB에서 조회하여 엔티티를 만들어서 1차 캐시에 넣는다.
    (이 때 스냅샷에 해당 엔티티를 저장한다.)
4. 1차 캐시에서 조회하여 엔티티를 반환한다.

(2) 등록

1. 이미 1차 캐시에 존재하는지 확인한다.
2. 있다면 넘어간다.
3. 없다면 1차 캐시에 키값, 엔티티를 저장하고 쓰기 지연 저장소에 SQL문을 등록한다.

(3) 수정

1. UPDATE 쿼리문을 쓰기 지연 저장소에 SQL문을 등록한다.
2. flush 될 때 스냅샷과 엔티티 간 다른 값을 가지는게 있다면 update 문을 쓰기 지연 저장소에 추가한다.
3. flush 될 때 등록된 쿼리문들이 반영되면서 update가 진행된다.

(4) 삭제

## 3. 플러시

플러시: 데이터 베이스에 쓰기 지연 저장소의 SQL문을 전달하여 동기화 하는 과정

- Dirty Checking:

플러시 진행 시 엔티티와 해당 엔티티의 스냅샷을 비교한다.
만약 다른 필드 값이 조회될 경우 업데이트 SQL문을 쓰기 지연 저장소에 추가한다.

이때 쿼리문이 발생,
쿼리문은 전체 필드에 대해 업데이트를 하는 쿼리문을 사용

다른 필드에 대해서도 업데이트 하기 때문에 단점이 있지만 (필드가 30개 이상이 아니라면 성능상 단점이 크지 않다)
- 쿼리문을 캐시해서 재사용할 수 있는 장점
- 쿼리문을 새로 조합하지 않아도 된다는 장점

- 반드시 커밋 전에 플러시가 일어나야함, 그래서 JPA는  커밋 전에 플러시가 자동으로 진행되도록 함

## 4. 준영속

준영속 상태란 영속 상태였던 엔티티를 엔티티 컨텍스트가 관리하지 않는 상태라는 뜻

준영속 상태는 영속 상태의 이점을 전혀 받지 못한다.

- 영속성 컨텍스트 초기화 `clear()`
영속 상태였던 엔티티를 모두 준영속 상태로 변경,
쓰기 지연 SQL 저장소에 있던 쿼리문들을 전부 삭제

- 영속성 컨텍스트 종료 `close()`
영속상태였던 엔티티를 전부 삭제,
쓰기 지연 SQL 저장소에 있던 쿼리문들을 전부 삭제

- 병합 `merge()`
준영속 상태의 엔티티를 영속 상태로 다시 병합

- 이 경우 준영속 상태였던 엔티티와 (별도의 업데이트를 진행하지 않은 경우) 동일성을 갖지만, 동등성은 갖지 않는다.

다시 병합할 경우

1. 병합하기 위해 키값을 넘겨줌
2. 해당 키값으로 1차 캐시에서 다시 조회 (보통은 이 경우 다시 조회할 수 없음)
3. DB에서 해당 엔티티를 다시 조회
4. 조회한 상태를 스냅샷으로 저장
5. 조회한 상태를 1차 캐시에 저장
