## 7장: 고급 매핑

### 1. 상속 관계 매핑

객체의 상속 구조와 데이터베이스의 슈퍼타입 / 서브타입 관계의 매핑

### (1) 방법
슈퍼타입 / 서브타입 (논리 모델) -> 테이블 (물리 모델)
1. 각각의 테이블로 변환 후 조회 시 JOIN 사용 - JPA의 JOIN 전략
2. 통합 테이블로 변환 - JPA의 단일 테이블 전략
3. 서브타입 테이블로 변환 - JPA의 구현 클래스마다 테이블 전략

### (1) JOIN 전략

- 엔티티 각각을 모두 테이블로 만든 뒤,  
자식 테이블이 부모 테이블의 기본키를 받아 기본키 + 외래키로 사용하는 전략
- 조회할 때 JOIN을 사용

- DTYPE이라는 것을 사용하여 타입의 개념이 없는 테이블을 구분하는 컬럼 추가

- 설정 방법: 
    1. 부모테이블에 `@Inheritance(strategy= InheritanceType.JOINED)`로 매핑 전략 지정
    2. 부모테이블에 `@DiscriminatorColumn(name="DTYPE")`로 구분 컬럼을 지정, 이 컬럼으로 자식 테이블 구분 가능
    3. 자식 테이블에 `@DiscriminatorValue("DTYPE값")`로 구분 칼럼을 입력할 값을 지정
    4. (선택) 자식 테이블에 `@PrimaryKeyJoinColumn(name = "지정할 이름")`으로 기본키 컬럼명 수정 가능,  
        미지정 시 부모 테이블 pk 컬럼명 그대로 사용

> JPA 표준 명세는 구분 컬럼을 사용하도록 하지만,
> 하버네이트를 포함한 구현체는 구분 컬럼 `@DiscriminatorColumn(name="DTYPE")` 없이도 동작한다.

#### 장점:
1. 테이블이 정규화 된다.
2. 외래키 참조 무결성 제약조건을 사용할 수 있다.
3. 저장 공간을 효율적으로 사용한다,

#### 단점:
1. 조회할 때 JOIN이 많이 사용되므로 성능 저하를 유발할 수 있다.
2. 조회 쿼리가 복잡하다.
3. 데이터를 등록할 INSERT SQL을 두 번 실행한다.

### (2) 단일 테이블 전략

테이블을 하나만 사용한다.  
구분 컬럼으로 어떤 자식 데이터가 저장되었는지 구분한다.

- 설정 방법: 
    1. 부모테이블에 `@Inheritance(strategy= InheritanceType.SINGLE_TABLE)`로 매핑 전략 지정
    2. 부모테이블에 `@DiscriminatorColumn(name="DTYPE")`로 구분 컬럼을 **필수로** 지정 (이 테이블 하나에 모든 것을 통합하기 때문)
    3. 자식 테이블에 `@DiscriminatorValue("DTYPE값")`로 구분 칼럼을 입력할 값을 지정
        ("DTYPE값" 미지정 시 엔티티 이름을 그대로 사용)

#### 장점:
1. 조인이 필요 없으므로 조회 성능이 빠르다
2. 조회 쿼리가 단순하다

#### 단점:
1. 자식 엔티티가 매핑한 컬럼은 모두 null을 허용해야한다.  
    > 부모 엔티티에 여러 자식 엔티티를 경우에 그 중 하나의 자식 엔티티만 저장하면 나머지 자식 엔티티의 컬럼은 사용되지 않으므로 null이 입력되기 때문이다.
2. 단일 테이블에 모든 것을 저장하므로 테이블이 커질 수 있다. 상황에 따라서는 조회 성능이 오히려 느려질 수 있다.

### (3) 구현 클래스마다 테이블 전략

자식 엔티티마다 테이블을 만든다. (비추천)
자식 테이블 각각에 필요한 컬럼이 모두 있다.

- 설정 방법: 
    1. 부모테이블에 `@Inheritance(strategy= InheritanceType.TABLE_PER_CLASS)`로 매핑 전략 지정
    2. 구분 컬럼을 사용하지 않는다.

#### 장점:
1. 서브 타입을 구분해서 처리할 때 효과적이다.
2. not null 제약 조건을 사용할 수 있다.

#### 단점:
1. 여러 자식 테이블을 함께 조회할 때 성능이 느리다 (SQL의 UNION 사용)
2. 자식 테이블을 통합해서 쿼리하기 어렵다

## 2. @MappedSuperClass

부모 클래스를 테이블과 매핑하지 않고, **상속 받는 자식 클래스에게 매핑 정보만 제공**하는 클래스

#### 상속받은 entity

- `@AttributeOverride(name="지정할 이름" , column = @Column(name="변경할 컬럼명"))`: 상속받은 속성의 컬럼명 수정
- `@AttributeOverrides({@AttributeOverride(), @AttributeOverride(), ... })`: 상속받은 속성들의 컬럼명 수정

#### 특징
- @MappedSuperClass로 지정된 클래스는 entity가 아니므로 em.find() 나 JPQL을 사용할 수 없다.
- 직접 생성하여 사용할 일은 거의 없으므로 abstract 클래스로 만드는 것을 권장한다.

## 3. 복합키와 식별 관계 매핑

프로젝트 진행 시 비식별관계를 사용하고, 기본키는 Long 타입의 대리키를 사용하자.

식별관계:
    - 장점: 기본키 인덱스를 활용하기 좋고, 특정 상황에 하위 테이블만으로도 조인 없이 검색 가능
    - 단점: 테이블 구조가 유연하지 못하다.
선택적 비식별관계:
    - Null을 허용하므로 조인할 때 외부 조인을 사용해야한다.
필수적 비식별관계:
    - Not null로 항상 관계가 있다는 것을 보장하므로 내부 조인만 사용해도 된다.

- 식별 관계: 부모 테이블의 기본키를 **자식 테이블의 기본키&외래키**로 사용하는 관계
- 비식별 관계: 부모 테이블의 기본키를 **자식 테이블의 외래키로만** 사용하는 관계

> 외래키에 NULL 허용 여부에 따라 분류된다.
> - 필수적 비식별 관계: 외래키는 not null, 연관관계를 필수로 맺어야한다.
> - 선택적 비식별 관계: 외래키 null 가능, 연관관계를 선택적으로 맺을 수 있다.

### (1) 비식별 관계 & 복합키

JPA에서 식별자를 둘 이상 사용하려면 별도의 식별자 클래스가 필요하다.
JPA는 영속성 컨텍스트에서 엔티티를 구분하기 위해 식별자를 사용, 식별자를 구분하기 위해 equals, hashCode를 사용하여 동등성 비교를 한다.
- 식별자 필드가 1개: 자바의 기본 타입 사용
- 식별자 필드가 2개 이상: 별도의 식별자 클래스 생성 및 equals, hashCode 구현 필요

JPA가 복합키를 지원하는 방법: @IdClass, @EmbeddedId

#### @IdClass

1. 엔티티에 @IdClass 를 사용하여 식별자 클래스를 지정한다.
    - 엔티티에는 매핑될 키 필드가 존재해야한다.
    - 식별자 클래스의 필드명과 일치해야한다.
2. 식별자 클래스
    - Serializable 인터페이스를 구현해야한다. (왜?)
    - 속성명이 엔티티에서 사용하는 식별자의 속성명이 같아야한다.
    - 기본 생성자가 있어야한다.
    - 식별자 클래스는 public이어야한다.

- 엔티티 복합키 설정은 각각 필드를 하나씩 저장하면 된다. (식별자 클래스 사용 불필요)
- 영속성 컨텍스트를 엔티티를 등록하기 전 저장된 복합키값을 통해 식별자 클래스를 생성하고, 영속성 컨텍스트의 키로 사용한다.
- 복합키를 통해 엔티티를 조회할 수 있다.

- 자식 클래스 추가 시 왜래키 컬럼 지정을 @JoinColumns({@JoinColumn(), @JoinColumn(), ..})로 매핑할 수 있다.

#### @EmbeddedId

1. 엔티티에서 식별자 클래스를 직접 사용하고, @EmbeddedId를 적어주면 된다.
2. 식별자 클래스
    - @Embeddable 애노테이션을 붙여준다.
    - Serializable 인터페이스를 구현해야한다.
    - 속성명이 엔티티에서 사용하는 식별자의 속성명이 같아야한다.
    - 기본 생성자가 있어야한다.
    - 식별자 클래스는 public이어야한다.

> 복합키에는 @GenerateValue를 사용할 수 없다.
> @EmbeddedId가 좀 더 객체 지향적이고 중복도 없지만, 특정 상황에 JPQL이 조금 더 길어질 수 있다.

### (2) 식별 관계 & 복합키

식별 관계에서 자식 테이블은 부모테이블의 기본키를 포함해서 복합키를 구성해야한다.
@IdClass, @EmbeddedId 를 사용하여 식별자를 매핑한다.

#### @IdClass

- 단, 외래키도 복합키로 매핑해야하므로 식별자 클래스에 외래키의 필드명을 선언하여 매핑
- 기본키에 @Id로 매핑, 외래키 매핑인 @JoinColum, 연관관계 매핑인 @ManyToOne 사용

#### @EmbeddedId

- 식별관계 구성 시 기본키에 @MapsId를 사용
- 엔티티에서 기존 복합키에는 @EmbeddedId가 붙어있을 것임, 여기에 엔티티의 외래키에 @MapsId("식별자 클래스의 필드명")를 사용하여 기본키로 구성
- 식별자 클래스에는 기본키를 포함한 복합키들의 필드들을 선언해야한다.

### (3) 일대일 식별관계

자식 테이블의 기본 키 값으로 부모 테이블의 기본 키 값만 사용

- 부모 테이블의 기본키가 복합키가 아니라면 자식 테이블의 기본키는 복합키로 구성하지 않아도 된다.
- 자식 테이블의 매핑 컬럼에 @MapsId만 사용하고 속성 값은 비워두면 된다.

## 3. 조인 테이블

데이터베이스 테이블의 연관관계를 설계하는 방법

### (1) 조인 컬럼 (외래키 컬럼) 사용

선택적 비식별관계:
    - Null을 허용하므로 조인할 때 외부 조인을 사용해야한다.
필수적 비식별관계:
    - Not null로 항상 관계가 있다는 것을 보장하므로 내부 조인만 사용해도 된다.

### (2) 조인 테이블 사용

별도의 테이블을 사용해서 연관관계를 관리하는 방식,
각 테이블에 연관관계를 관리하기 위한 외래키 컬럼이 없다.

- `@JoinColumn` : 조인 컬럼 매핑
- `@JoinTable` : 조인 테이블 매핑

#### 일대일 조인 테이블

- 조인 테이블 외래키 컬럼 각각에 2개의 유니크 제약 조건 필요
- 부모 엔티티: 연관관계의 주인에서 매핑하는 필드에 `@JoinTable`를 추가하여 조인 테이블 사용

#### 일대다 조인 테이블, 다대일 조인 테이블

- 다 쪽에서 연관관계를 정의함, 조인 테이블도 다 쪽에서 정의

#### 다대다 조인 테이블

- 원래 @ManyToMany면 기본 규칙에 따라 조인 테이블 생성해줌
- 컬럼명, 테이블명 지정을 원하면 정의

### (3) 엔티티 하나에 여러 테이블 매핑

- 설정 방법: `@SecondaryTable`, `@SecondaryTables({@SecondaryTable(), @SecondaryTable(), ... })`

- `@SecondaryTable`를 통해 추가로 테이블 매핑 가능
- 물리적으로는 2개의 테이블이지만 논리적으로는 하나의 entity
- pk 값은 공유한다.
- @Column(table = "추가로 매핑한 테이블명")을 지정하지 않으면 엔티티에 @Table(name="")로 지정한 기본 테이블에 매핑된다.
- 항상 두 테이블을 조회하므로 최적화가 어렵다.
