## 1장: JPA 소개

## 1. 개요

데이터베이스는 데이터 중심의 구조로 데이터를 다룬다.
자바는 객체로 데이터를 다룬다.

자바로 작성한 **객체**를 데이터베이스의 **테이블 구조**에 맞게 저장할 수 있도록 **변환해주는 중간 계층**이 JPA 이다.

## 2. JPA 개념

자바 진영의 ORM 기술 표준이다.

### 2.1 ORM
Object-Relational Mapping는 이름 그대로 객체와 관계형 데이터 베이스를 매핑한다는 것이다.

객체와 테이블을 매핑하여 패러다임 불일치 문제를 해결하준다.  
개발자는 매핑 방법에 대해서만 ORM 프레임워크에게 알려주면 된다.

### 2.2 장점

**1. 생산성**

개발자가 원래 했어야할 SQL문 작성, 반복적인 JDBC API 사용을 대신 해준다.

**2. 유지보수**

엔티티에 필드 하나가 추가되면 개발자는 다음과 같은 일을 했어야했다.

- 관련된 SQL문 모두 수정
- 결과를 매핑하기 위한 JDBC API 코드 변경

그러나 JPA를 사용하면 해당 코드를 대신 처리해준다.

**3. 성능**

애플리케이션과 데이터베이스 사이에 계층이 하나 더 있기 때문에 최적화 관점에서 시도해볼 수 있는 것들이 많다.

- 예를 들어 같은 트랜잭션 안에서 객체를 두번 조회하는 경우, 두번째에는 이미 조회한 객체를 재사용한다.

**4. 데이터 접근 추상화와 벤더 독립성**

관계형 데이터베이스는 같은 기능도 벤더마다 사용법이 다른 경우가 많다.

추상화된 데이터 접근 계층을 제공해서 JAP 는 특정 데이터베이스 기술에 종속되지 않도록 한다.


**5. 패러다임 불일치 문제 해결**

JPA는 상속, 연관관계, 객체 그래프 탐색, 비교하기와 같은 패러다임 불일체 문제를 해결해준다.

## 3. JPA 가 해결하는 패러다임의 불일치

**1. 관계를 가지는 데이터를 함께 조회**

JPA: 데이터를 객체로 관리하는 자바 => SQL문

- **객체**는 관계를 가지는 객체를 함께 조회하고 싶다면 해당 객체의 **참조값**을 알고 있어야한다.
- **DB**는 연관관계에 있는 데이터를 함께 조회하고 싶다면 해당 데이터의 **key값**을 알고 있어야한다.

- **[상속]의 경우** 

    *상속은 관계형 DB에는 없는 개념이다.*
    - JPA의 `find()` 메서드를 사용하면 상속 관계에 있는 데이터를 key값을 통해 `join`한다.

- **[연관관계]의 경우**
    - 연관관계를 맺을 객체의 **참조값을** 넣어두면 JPA가 **해당 객체의 PK를 꺼내서** `join`한다.

**2. 객체 그래프 탐색**

- "SQL을 직접 다루면 처음 실행하는 SQL에 따라 객체 그래프를 어디까지 탐색할 수 있는지 정해진다."
- 객체는 그래프탐색을 할 수 있다.  
(그래프 탐색: 참조를 사용하여 연관된 객체를 찾는 것)

JPA를 사용하면 객체 그래프 탐색이 가능하다.  
JPA는 연관된 객체를 **사용하는 시점**에 `SELECT SQL`을 실행한다.  
(지연로딩 : 실제 사용하는 시점까지 조회를 미룬다)

*연관된 객체를 함께 즉시 조회할지, 지연로딩을 사용할지는 설정 가능하다.*


**3. 비교**

- 객체는 다음과 같은 비교 방법을 사용한다.  
    * **동일성 비교 (인스턴스 주소값)**
    * 동등성 비교(객체 내부의 값)
- DB는 PK값으로 row를 비교한다.

- jpa를 사용하지 않을 경우, 기본키 값으로 객체를 **두번 조회** 후 **동일성 비교** 를 진행한다면 결과는 `false`이다.  
객체 측면에서 **서로 다른 인스턴스**이기 때문이다.

- JPA를 사용한다면 `true`이다.  
같은 트랜잭션일 때 동일한 객체가 조회되는 것을 보장한다.  
(이는 마치 자바 컬렉션에 저장된 객체를 조회하는 것처럼 해준다.)
