# JPA 소개

## SQL을 직접 다룰 때 발생하는 문제점

### 반복, 반복 그리고 반복

데이터베이스는 객체 구조가 아니라 데이터 중심 구조로 되어있기 때문에 객체를 그대로 조회하거나 저장할 수 없다. 따라서 개발자가 SQL과 JDBC를 이용해서 객체를 데이터 형태로 바꾸고 다시 객체로 변환하는 작업을 직접 구현해야 한다.

문제는 테이블이 많아질수록 이러한 변환 코드도 반복해서 작성해야 하기 때문에 생산성이 떨어지는 문제가 발생한다.

### SQL에 의존적인 개발

**엔티티를 신뢰할 수 없다**

`Member`나 `Team`처럼 비즈니스 요구사항을 모델링한 객체를 엔티티라 하는데 SQL에 모든 것을 의존하는 상황에서는 개발자들이 엔티티를 신뢰하고 사용할 수 없다.

**진정한 의미의 계층 분할이 어렵다**

버그가 발생하거나 수정 사항이 생겼을 때는 DAO를 열어서 직접 어떤 SQL이 실행되고 어떤 객체들이 함께 조회되는지 일일이 확인해야 한다.

**SQL에 의존적인 개발을 피하기 어렵다**

물리적으로 SQL과 JDBC API를 데이터 접근 계층에 숨기는 데 성공했을지는 몰라도 논리적으로는 엔티티와 아주 강한 의존관계를 가지고 있다.

이러한 강한 의존관계 때문에 회원을 조회할 때는 물론이고 회원 객체에 필드를 하나 추가할 때도 DAO의 CRUD 코드와 SQL 대부분을 변경해야 하는 문제가 발생한다.

### JPA와 문제 해결

JPA를 사용하면 객체를 데이터베이스에 저장하고 관리할 때 가발자가 직접 SQL을 작성하는 것이 아니라 JPA가 제공하는 API를 사용하면 된다.

## 패러다임의 불일치

관계형 데이터베이스에는 데이터 중심으로 구조화되어 있고 집합적인 사고를 요구한다. 또한, 객체지향에서의 추상화, 상속, 다형성과 같은 개념이 없다.

객체와 관계형 데이터베이스는 지향하는 목적이 서로 다르기 때문에 둘의 기능과 표현 방법도 다른데 이를 객체와 관계형 데이터베이스의 패러다임 불일치 문제라고 한다.

### 상속

객체는 상속이라는 기능을 가지고 있지만 테이블은 상속이라는 기능이 없다.

<img width="421" height="235" alt="스크린샷 2026-03-18 오후 2 04 19" src="https://github.com/user-attachments/assets/cbf34e6b-16e6-48b7-a91b-19c9749d6ce1" />

슈퍼타입 서브타입 관계를 사용하면 객체 상속과 가장 유사한 형태로 테이블을 설계할 수 있다.

<img width="409" height="206" alt="스크린샷 2026-03-18 오후 2 14 32" src="https://github.com/user-attachments/assets/cca0c595-71da-4db3-9ebb-087488e8d8f9" />

JDBC API를 사용하면 부모 객체에서 부모 데이터만 꺼내 SQL을 작성하고 자식 객체에서 자식 데이터만 꺼내서 SQL을 작성해야 하기 때문에 작성해야 할 코드량이 많다.

JPA는 상속과 관련된 패러다임의 불일치 문제를 해결해준다. 개발자는 마치 자바 컬렉션에 객체를 저장하듯이 JPA에게 객체를 저장하면 된다.

JPA를 사용해서 저장할 때는 persist() 메소드를 사용해서 객체를 저장하면 된다.

``` java
jpa.persist(album);
```

JPA는 다음 SQL을 실행해서 객체를 ITEM, ALBUM 두 테이블에 나누어 저장한다.

``` sql
INSERT INTO ITEM ...
INSERT INTO ALBUM ...
```

Album 객체를 조회할 때는 find() 메소드를 사용해서 객체를 조회하면 되고 JPA는 ITEM과 ALBUM 두 테이블을 조인해서 필요한 데이터를 조회하고 그 결과를 반환한다.

### 연관관계

객체는 참조를 사용해서 다른 객체와의 연관관계를 가지고 참조에 접근해서 연관된 객체를 조회한다.

반면에 테이블은 외래 키를 사용해서 다른 테이블과 연관관계를 가지고 조인을 사용해서 연관된 테이블을 조회한다.

<img width="534" height="308" alt="스크린샷 2026-03-18 오후 2 15 24" src="https://github.com/user-attachments/assets/b8361fa3-de10-4c56-81de-95b07216f174" />

**객체를 테이블에 맞추어 모델링**

```java
class Member {

    String id;
    Long temaId;
    String username;
}

class Team {

    Long id;
    String name;
}
```

객체를 테이블에 맞추어 모델링하면 객체를 테이블에 저장하거나 조회할 때는 편리하지만 외래 키의 값을 그대로 보관하는 teamId 필드에는 문제가 있다. 객체는 연관된 관계를 보관해야 참조를 통해 연관된 객체를 찾을 수 있기 때문이다. 외래 키까지 관계형 데이터베이스가 사용하는 방식에 맞추면 객체지향의 특징을 잃어버리게 된다.

```java
Team team = member.getTeam();
```

**객체지향 모델링**

객체는 참조를 통해서 관계를 맺는다.

```java
class Member {

    String id;
    Team team;
    String username;

    Team getTeam() {
        return team;
    }
}

class Team {

    Long id;
    String name;
}
```

외래 키의 값을 그대로 보관하는 것이 아니라 연관된 Team의 참조를 보관한다. 그런데 객체지향 모델링을 사용하면 객체를 테이블에 저장하거나 조회하기 쉽지 않다. 객체 모델은 외래 키가 필요 없고 단지 참조만 있으면 되며 테이블은 참조가 필요 없고 외래 키만 있으면 되기 때문이다.

결국, 개발자가 중간에서 변환 역할을 해야 한다.

**JPA와 연관관계**

JPA는 연관관계와 관련된 패러다임의 불일치 문제를 해결한다. 개발자는 회원과 팀의 관계를 설정하고 회원 객체를 저장하면 JPA가 적절한 SQL을 데이터베이스에 전달한다. 객체를 조회할 때도 마찬가지다.

### 객체 그래프 탐색

객체에서 회원이 소속된 팀을 조회할 때는 참조를 사용해서 연관된 팀을 찾으면 되는데 이를 객체 그래프 탐색이라고 한다.

```java
Team team = member.getTeam();
```

<img width="542" height="266" alt="스크린샷 2026-03-18 오후 2 33 16" src="https://github.com/user-attachments/assets/b2231d22-bab7-4e87-b8b1-a9bf090b0e76" />

```sql
SELECT M.*, T.*
    FROM MEMBER M
    JOIN TEAM T ON M.TEAM_ID = T.TEAM_ID
```

MemberDAO에서 객체를 조회할 때 위의 SQL을 실행해서 회원과 팀에 대한 데이터만 조회했다면 member.getOrder();은 null이 된다.

SQL을 직접 다루면 처음 실행하는 SQL에 따라 객체 그래프를 어디까지 탐색할 수 있는지 정해진다.

```java
class MemberService {
    ...
    public void process() {

        Member member = memberDAO.find(memberId);
        member.getTeam();
        member.getOrder().getDelivery();
    }
}
```

memberDAO를 통해서 객체를 조회했지만 연관된 Team, Order, Delivery 객체 그래프를 탐색할 수 있는지는 코드만 보고는 예측이 불가하기 때문에 DAO를 통해서 확인해야 한다.
결국, MemberDAO에 회원을 조회하는 메소드를 상황에 따라 여러 벌 만들어서 사용해야 한다.

```java
memberDAO.getMember();
memberDAO.getMemberWithTeam();
memberDAO.getMemberWithOrderWithDelivery();
```

**JPA와 객체 그래프 탐색**

JPA를 사용하면 객체 그래프를 마음껏 탐색할 수 있다.

JPA는 연관된 객체를 사용하는 시점에 적절한 SQL을 실행하기 때문에 연관된 객체를 신뢰하고 조회할 수 있다. 이 기능은 실제 객체를 사용하는 시점까지 데이터베이스 조회를 미룬다고 해서 **지연 로딩**이라고 한다.

```java
// 처음 조회 시점에 SELECT MEMBER SQL
Member member = jpa.find(Member.class, memberId);

Order order = member.getOrder();
order.getOrderDate(); // Order을 사용하는 시점에 SELECT ORDER SQL
```

### 비교

데이터베이스는 기본 키의 값으로 각 row를 구분하며 객체는 동일성 비교와 동등성 비교가 있다.
- 동일성 비교는 == 비교로 객체 인스턴스의 주소 값을 비교한다.
- 동등성 비교는 equals() 메소드를 사용해서 객체 내부의 값을 비교한다.

```java
String memeberId = "100";
Member member1 = memberDAO.getMember(memberId);
Member member2 = memberDAO.getMember(memberId);

member1 == member2; // false
```

기본 키 값이 같은 회원 객체를 두 번 조회했으나 객체 측면에서 볼 때 둘은 다른 인스턴스이기 때문에 동일성 비교에서 false가 반환된다.

**JPA와 비교**

JPA는 같은 트랜잭션일 때 같은 객체가 조회되는 것을 보장한다.

```java
String memeberId = "100";
Member member1 = jpa.find(Member.class, memberId);
Member member2 = jpa.find(Member.class, memberId);

member1 == member2; // true
```

### 정리

패러다임의 차이를 극복하려고 개발자가 너무 많은 시간과 코드를 소비한다. JPA는 패러다임의 불일치 문제를 해결해주고 정교한 객체 모델링을 유지하게 도와준다.

## JPA란 무엇인가?

JPA는 자바 진영의 ORM 기술 표준으로 애플리케이션과 JDBC 사이에서 동작한다.

ORM은 객체와 관계형 데이터베이스를 매핑한다는 뜻이며, ORM 프레임워크는 객체와 테이블을 매핑해서 패러다임의 불일치 문제를 개발자 대신 해결해준다.

<img width="630" height="358" alt="스크린샷 2026-03-18 오후 2 52 47" src="https://github.com/user-attachments/assets/1871e7d6-4c47-419b-b109-5fed63008d6a" />

jpa.persist(member);의 코드를 사용해서 객체를 저장할 수 있다.

<img width="642" height="373" alt="스크린샷 2026-03-18 오후 2 52 56" src="https://github.com/user-attachments/assets/171d0b52-21d1-49b1-8925-387aa83e9f8f" />

Member member = jpa.find(memberId);의 코드를 사용해서 객체를 조회할 수 있다.

ORM 프레임워크를 통해 객체 측면에서는 정교한 객체 모델링이 가능하고 관계형 데이터베이스는 데이터베이스에 맞게 모델링 하면 된다. 매핑 방법만 ORM 프레임워크에만 알려주면 알아서 처리하기 때문에 관계형 데이터베이스를 사용해도 객체지향 애플리케이션 개발에 집중할 수 있다.

### JPA 소개

JPA는 자바 ORM 기술에 대한 API 표준 명세이다. JPA라는 표준 덕분에 특정 구현 기술에 대한 의존도를 줄일 수 있고 다른 구현 기술로 손쉽게 이동할 수 있는 장점이 있다.

### 왜 JPA를 사용해야 하는가?

**생산성**

- JPA를 사용하면 자바 컬렉션에 객체를 저장하듯이 JPA에 저장할 객체를 전달하면 JPA가 SQL을 작성하고 JDBC API를 사용하는 반복적인 작업을 처리해준다.

**유지보수**

- SQL을 직접 다루면 엔티티에 필드를 하나만 추가해도 관련된 등록, 수정, 조회 SQL과 결과 매핑 JDBC API 코드를 모두 변경해야 했으나 JPA를 사용하면 JPA가 대신 처리해야 하기 때문에 수정해야 할 코드가 줄어든다.

**패러다임의 불일치 해결**

- JPA는 상속, 연관관계, 객체 그래프 탐색, 비교하기와 같은 패러다임의 불일치 문제를 해결해준다.

**성능**

- JPA는 애플리케이션과 데이터베이스 사이에서 다양한 성능 최적화 기회를 제공한다. 
- 트랜잭션 안에서 같은 회원을 두 번 조회하는 코드에서는 JPA를 사용하면 회원을 조회하는 SQL을 한 번만 데이터베이스에 전달하고 두 번째는 조회한 회원 객체를 재사용하기 때문에 성능이 향상된다.

**데이터 접근 추상화와 벤더 독립성**
- 관계형 데이터베이스는 같은 기능도 벤더마다 사용법이 다른 경우가 많다.
- JPA는 애플리케이션과 데이터베이스 사이에 추상화된 데이터 접근 계층을 제공해서 애플리케이션이 특정 데이터베이스 기술에 종속되지 않도록 한다.

**표준**
JPA는 자바 진영의 ORM 기술 표준으로 다른 구현 기술로 손쉽게 변경할 수 있다.
