## 5장: 연관관계 매핑 기초

연관관계 매핑은 **객체의 참조**와 **테이블의 외래키**의 매핑을 의미한다.

| 구분         | 객체                                                         | 테이블                           |
| ---------- | ---------------------------------------------------------- | ----------------------------- |
| 연관관계 저장 방식 | 연관 관계를 가질 객체의 참조값을 가지는 필드를 객체 내부에 저장하고, 이 참조값으로 연관관계를 맺는다. | 외래키(FK)로 연관관계를 맺는다.        |
| 연관관계 탐색 방식 | 참조를 통한 연관관계 탐색(객체 그래프 탐색)                           | 외래키를 사용한 연관관계 탐색(JOIN) |
| 관계의 방향성    | 단방향                                                 | 양방향 (방향 개념 없음)                    |

> 객체의 참조를 통한 연관 관계는 언제나 단방향이다.  
> 객체 간의 양방향 연관관계는 2개의 단방향 관계를 의미한다.

## 1. 단방향 연관관계

### (1) 단방향 매핑 예제 코드

Team과 Member는 일대다 관계이다.
- Team은 0개 이상의 Member를 가진다.
- Member는 1개 이하의 팀에 속한다.

```java
@Entity
public class Team{

    @Id
    @Column(name = "TEAM_ID")
    private String id;

    // 단방향이므로 Member에 대한 참조(List<Member>)가 없음
    ...
}

@Entity
public class Member{

    @Id
    @Column(name = "MEMBER_ID")
    private String id;

    ...

    @ManyToOne // 매핑 정보
    @JoinColumn(name = "TEAM_ID") // MEMBER 테이블의 FK 컬럼명을 "TEAM_ID"로 지정
    private Team team;

    public void setTeam(Team team) {
        this.team = team;
    }
}
```


### (2) 매핑 시 사용되는 속성

#### @JoinColumn 주요 속성

@JoinColumn 생략 시 외래키 찾을 때 기본 전략을 사용  
(`현재 테이블의 필드명 + _ + 참조 테이블의 pk 컬럼명)  
> 필드명: team  
> 참조 PK 컬럼명: TEAM_ID  
> 기본 FK 컬럼명: team_TEAM_ID  


- name: 매핑할 외래키 이름 (기본값: `현재 테이블의 필드명 + _ + 참조 테이블의 pk 컬럼명`)
- referencedColumnName: 외래키가 참조하는 대상 테이블의 컬럼명 (기본값: `기본키 컬럼명`)
- foreignKey(DDL): 외래키 제약 조건 지정
- unique, nullable, insertable, updatable, columnDefinition, table: @Column의 속성과 동일

#### @ManyToOne 주요 속성

- optional: 연관된 엔티티가 반드시 존재해야하는지 (기본값: true)
- fetch: 글로벌 패치 전략 사용
- cascade: 영속성 전이 기능 사용
- targetEntity: 연관된 엔티티의 타입 정보를 알려줌, 제네릭이 없는 경우 사용
    > - `List members` -> 타입을 알 수 없다, Memeber 타입이라고 targetEntity 속성을 통해 지정 필요  
    > - `List<Memeber> members` 라면 타입을 알 수 있으므로 필요 없음

## 2. 양방향 연관관계

객체의 참조는 둘인데, 데테이블의 외래키는 하나이다.  
**관리포인트를 하나로 모으기 위해** 테이블의 외래키를 관리하는 **연관관계의 주인**을 정한다.

양방향과 단방향의 차이는 반대방향으로 객체 그래프 탐색이 가능하다는 것이다.

| 구분        | 주인(Owner)                               | 주인이 아닌 쪽                              |
| --------- | --------------------------------------- | ------------------------------------- |
| 외래키 관리    | 외래키 관리 가능 (등록, 수정, 삭제)                  | 외래키 관리 불가                             |
| 데이터 변경 권한 | 연관관계 변경 가능                              | 읽기만 가능                                |
| 관계 기준     | 다대일, 일대다 관계에서 ‘다’쪽(@ManyToOne)이 외래키를 가짐 | 주인이 아닌 쪽은 외래키를 가지지 않음                 |
| 설정 방법     | 별도 설정 없이 외래키를 가진 쪽이 주인이 됨               | mappedBy 속성에 주인 엔티티의 필드명을 지정하여 주인을 명시 |

### (1) 연관관계 설정 시 주의점

- 주인인 쪽에 연관관계 설정: 데이터베이스에 반영 O
- 주인이 아닌 쪽에만 연관관계 설정: 데이터베이스에 반영 X

```java
member1.setTeam(team1); // 주인
team1.getMembers().add(member1); // 주인이 아닌쪽
```

- **객채 관점**에서 보면 **양쪽 모두**에 값을 설정해주는 것이 좋다.  
    _(JPA를 사용하지 않는 순수 객체 상태에서도 동작하도록)_

- **양쪽 모두 값을 설정해주지 않을 때 문제점:** 
    - 주인인 member에만 연관관계를 수정하고 DB에 반영하더라도, 동일한 영속성 컨텍스트 내에는 이미 로딩된 Team 객체의 members 리스트가 존재한다.
    - 따라서 `team1.getMembers()`를 호출하게 된다면 DB와는 다른 데이터를 조회하게 된다.

```java
@Entity
public class Team{

    @Id
    @Column(name = "TEAM_ID")
    private String id;

    // "Member 엔티티에 있는 team 필드에 의해 매핑되었다"
    @OneToMany(mappedBy = "team") 
    private List<Member> members = new ArrayList<>();

}

@Entity
public class Member{

    @Id
    @Column(name = "MEMBER_ID")
    private String id;

    @ManyToOne
    @JoinColumn(name = "TEAM_ID")
    private Team team;

    public void setTeam(Team team) {
        // 기존 팀과의 관계 제거
        if (this.team != null){
            this.team.getMembers().remove(this);
        }
        this.team = this;
        team.getMembers().add(this);
    }
}
```

**@JoinColumn**은 DB 컬럼명을 찾고, **mappedBy**는 매핑할 객체 필드명을 찾는다
